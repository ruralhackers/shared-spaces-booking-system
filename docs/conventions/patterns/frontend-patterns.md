# Frontend Patterns

This document defines conventions for the client apps and the shared UI package:

- `apps/webapp` — Vite SPA (React 19 + TanStack Router + shadcn/ui + Tailwind v4).
- `apps/mobile` — Expo 54 (React Native + expo-router + NativeWind).
- `packages/ui` — cross-platform design tokens, CVA variants, and `cn()`.

The frontend is a thin delegate — domain logic lives in `packages/` and is hosted by `apps/api` (see [apps/api](../../../apps/api)).

## Shared Principles

### Stay on the tRPC Boundary

- Every data read/write goes through `@dfs/api` procedures via the tRPC client.
- Mutations must invalidate the relevant queries on success (TanStack Query `invalidateQueries`).
- No direct Prisma, repository, or `@dfs/<context>/domain` imports from frontend code.
- No business invariants in the UI — call an application service; let it enforce the rule and surface the `DomainError`.

```typescript
// ✅ Good
const { mutate: createUser } = api.users.create.useMutation({
  onSuccess: () => utils.users.list.invalidate(),
})

// ❌ Bad — Prisma or repository in the UI
import { prisma } from '@dfs/database'
const users = await prisma.user.findMany()
```

### Never Fetch With `fetch()`

All HTTP goes through the tRPC client. It wires tracing, auth, and error translation automatically. Raw `fetch()` bypasses those guarantees.

### Target the `apps/api` Origin

Both web and mobile point their tRPC client and auth client at `apps/api` (default `http://localhost:4000`). Never re-host the router in the web app — the backend is a single, standalone Elysia process and both clients are equal.

## `apps/api` — Bun + Elysia

A thin HTTP shell around `@dfs/api` (tRPC router) and `@dfs/auth` (better-auth).

```
apps/api/src/
├── index.ts      # Elysia + CORS + mounts
├── env.ts        # @t3-oss/env-core validation
├── auth.ts       # createAuthConfig({...}) from @dfs/auth
├── context.ts    # createTRPCContext({ headers })
└── db.ts         # Prisma client singleton (dev-safe)
```

### Mount recipe

```typescript
import { appRouter } from '@dfs/api'
import { cors } from '@elysiajs/cors'
import { fetchRequestHandler } from '@trpc/server/adapters/fetch'
import { Elysia } from 'elysia'

const app = new Elysia()
  .use(cors({ origin: env.CORS_ORIGINS.split(','), credentials: true }))
  .all('/api/auth/*', ({ request }) => auth.handler(request))
  .all('/api/trpc/*', ({ request }) =>
    fetchRequestHandler({
      endpoint: '/api/trpc',
      req: request,
      router: appRouter,
      createContext: () => createTRPCContext({ headers: request.headers })
    })
  )
  .listen(env.PORT)
```

### What belongs here

- CORS, request plumbing, env validation.
- Mounting pre-existing handlers from `@dfs/api` and `@dfs/auth`.

### What does NOT belong here

- tRPC procedures (they live in `packages/api/src/routers/`).
- Domain logic of any kind.
- Prisma access outside of `db.ts` (which only exposes the client).

## `apps/webapp` — Vite SPA with TanStack Router

### File Layout

```
apps/webapp/src/
├── routes/          # TanStack file-based routes (__root.tsx, login.tsx, etc.)
├── routeTree.gen.ts # Generated by @tanstack/router-plugin (do not edit)
├── components/
│   └── ui/          # shadcn/ui primitives (copied recipes; consume @dfs/ui/variants)
├── features/        # Feature-level composition (page-sized UI for a domain concern)
├── hooks/           # Cross-feature hooks
├── stores/          # Zustand stores (persist for theme)
├── trpc/            # tRPC client + QueryClient wiring
├── lib/             # auth-client, cn(), pure utilities
├── styles/          # globals.css (Tailwind v4 @theme inline)
├── env.ts           # @t3-oss/env-core + import.meta.env.VITE_*
└── main.tsx         # Root: TRPCReactProvider + RouterProvider + Toaster
```

### Routing

File-based via `@tanstack/router-plugin/vite` (autoCodeSplitting). Each file in `routes/` becomes a route; `routeTree.gen.ts` is generated on build.

```tsx
// routes/dashboard.tsx — authenticated route with guard
export const Route = createFileRoute('/dashboard')({
  beforeLoad: async () => {
    const { data } = await authClient.getSession()
    if (!data?.user) throw redirect({ to: '/login' })
    return { user: data.user }
  },
  component: DashboardPage,
})
```

- Guards live in `beforeLoad` and throw `redirect(...)` — never conditional render inside the component for auth.
- Data loading in `loader` when the route needs tRPC data to paint; otherwise `useQuery` inside the component.

### tRPC client

```typescript
// src/trpc/react.tsx
const link = httpBatchStreamLink({
  transformer: SuperJSON,
  url: `${env.VITE_API_URL}/api/trpc`,
  fetch: (url, options) =>
    fetch(url, { ...(options as RequestInit), credentials: 'include' }),
})
```

- `credentials: 'include'` is required so the better-auth session cookie crosses the CORS boundary.
- URL comes from `env.VITE_API_URL` (validated via `@t3-oss/env-core`).
- Type: `import type { AppRouter } from '@dfs/api/client'` — the `/client` entry is tree-shakeable on the browser.

### Components

- Primitives are copied shadcn recipes under `components/ui/`. Do not fork ad-hoc.
- Variants come from `@dfs/ui/variants/*` — the CVA is shared with mobile. The component shell is per-platform.

```tsx
// src/components/ui/button.tsx
import { buttonVariants, type ButtonVariants } from '@dfs/ui/variants/button'
import { Slot as SlotPrimitive } from 'radix-ui'
import { cn } from '@/lib/utils'

type ButtonProps = React.ComponentProps<'button'> &
  ButtonVariants & { asChild?: boolean }

export function Button({ className, variant, size, asChild, ...props }: ButtonProps) {
  const Comp = asChild ? SlotPrimitive.Slot : 'button'
  return <Comp className={cn(buttonVariants({ variant, size, className }))} {...props} />
}
```

- Every query-backed section needs **loading**, **empty**, and **error** states.
- Accessibility: labels bound to inputs, `aria-*` for non-text controls, visible focus outlines, minimum 44×44px tap targets.

### Styling — Tailwind v4 + shadcn tokens

- Tailwind v4 via `@tailwindcss/vite`. No `tailwind.config.js` — `@theme inline` lives in `src/styles/globals.css`.
- Use theme tokens (`text-foreground`, `bg-muted`, `border`) — no hardcoded hex.
- Theme mode (light/dark) via Zustand `persist` + `localStorage`. FOUC is prevented by an inline `<script>` in `index.html` that reads `localStorage` before React mounts.
- `rem` for sizing; `px` only for borders and hairlines.

### Auth

- `authClient` from `better-auth/react` with `baseURL: env.VITE_API_URL` and `fetchOptions: { credentials: 'include' }`.
- Session cookies are set by `apps/api` with `SameSite=Lax` in dev and `SameSite=None; Secure` in prod across subdomains.

## `apps/mobile` — Expo 54 + expo-router + NativeWind

### File Layout

```
apps/mobile/
├── app/              # expo-router: route groups in (parens)
│   ├── (auth)/       # Pre-login routes
│   └── (tabs)/       # Post-login tab navigator
├── components/
│   └── ui/           # Local RN components (consume @dfs/ui/variants)
├── hooks/
├── providers/        # App-wide contexts (TRPCProvider, etc.)
├── lib/              # Pure utilities, auth, trpc config
├── global.css        # @tailwind base/components/utilities (NativeWind entry)
└── tailwind.config.js # Consumes @dfs/ui/tokens.tailwind for shared tokens
```

### Navigation

- Use `expo-router` conventions — `<Link>`, `useRouter`, route groups `(name)` — not manual stacks.
- Layouts (`_layout.tsx`) own the navigator configuration for their segment.
- Deep-linking and typed routes via generated types; don't hand-roll navigation params.

### NativeWind

- Write `className="bg-primary text-primary-foreground"` on `View`/`Text`/`Pressable`. Same class strings as web.
- Shared tokens come from `@dfs/ui/tokens.tailwind` (consumed in `tailwind.config.js`).
- For RN-only concerns (gestures, animated worklets), keep using `StyleSheet` or `react-native-reanimated` — NativeWind `className` alone is not enough.

### Platform Differences

- `Platform.OS` only when the behavior genuinely diverges. Prefer components that handle both.
- Safe-area via `SafeAreaProvider` and `useSafeAreaInsets` from `react-native-safe-area-context` — never hardcode top/bottom insets.
- Use `expo-image`, `expo-font`, `expo-router` primitives over bare React Native equivalents when available.

### Mobile-specific deps

These must be declared as direct deps of `apps/mobile` even though they are transitively pulled in by NativeWind, because Bun's isolated linker does not hoist them into reach of user code:

- `react-native-css-interop`
- `@babel/core`
- `@babel/plugin-transform-react-jsx`
- `babel-preset-expo`

## `packages/ui` — Cross-platform Foundations

### What lives here

- `tokens.ts` — design tokens (colors, radius, spacing, font).
- `tokens.tailwind.{ts,cjs}` — helpers to produce a `tailwind.config` `theme.extend` shape.
- `variants/*.variants.ts` — CVA variants (returns className strings, platform-agnostic).
- `lib/cn.ts` — `clsx` + `tailwind-merge`.
- `scripts/build-css-vars.ts` — emits `dist/tokens.css` for the web `@theme inline`.

### What does NOT live here

- Rendered components. The render tree for `<button>` (DOM) and `<Pressable>` (native) differs enough that sharing the component itself is a liability; sharing only the variant strings is robust and friction-free.
- Platform-specific logic (navigation, image loading, gestures).
- Icons — each app imports `lucide-react` (web) or `lucide-react-native` (mobile) locally.

### Adding a new shared variant

1. `packages/ui/src/variants/foo.variants.ts` — export `fooVariants` via `cva(...)`.
2. Add `"./variants/foo": "./src/variants/foo.variants.ts"` to `packages/ui/package.json` `exports`.
3. In each consumer (`apps/webapp/src/components/ui/foo.tsx` and `apps/mobile/components/ui/foo.tsx`) render the platform element with `className={fooVariants(...)}`.

### Adding a new token

1. Edit `packages/ui/src/tokens.ts`.
2. Run `bun run -F @dfs/ui build:tokens` to regenerate `dist/tokens.css` for the web.
3. `tailwind.config.js` on mobile picks up the change through `@dfs/ui/tokens.tailwind`.

## Data Fetching — tRPC + TanStack Query

### Queries

```typescript
const { data: users, isLoading, error } = api.users.list.useQuery(
  { limit: 20 },
  { staleTime: 30_000 }
)
```

Scope with stable keys. Let TanStack Query handle caching, background refetch, and retries.

### Mutations

```typescript
const utils = api.useUtils()
const { mutate } = api.users.create.useMutation({
  onSuccess: () => utils.users.list.invalidate(),
})
```

Always invalidate the queries that the mutation affects. Never mutate the local cache directly unless you have a strong reason (optimistic updates).

### Error Handling

Never swallow tRPC errors. Surface them through error boundaries, inline error UI, or a toast. Domain errors surface with a translated message and HTTP status; handle them explicitly.

## What to Avoid

- Business rules in components or hooks.
- `any` in component props or hook returns.
- Fetching via `fetch()` directly — always go through tRPC.
- Conditional hooks or hooks inside loops.
- Global mutable stores for data that the server owns — use the tRPC cache instead.
- Class components in new code — functional components with hooks only.
- Hosting the tRPC router or better-auth handler inside `apps/webapp` — they belong in `apps/api`.
- Rendering components from `@dfs/ui` with platform-coupled elements (DOM or RN) — only variants and tokens cross the boundary.
